# YeoboSay ver0.2 Conversational Call API Spec

Status: Draft  
Date: 2026-05-30  
Related plan: `docs/ver0.2/conversational_call_flow_plan.md`

## 1. Purpose

This document defines the API contract for ver0.2 conversational phone-call flow.

ver0.2 changes the MVP flow from manual "press record, speak, upload" interaction to a phone-call-like flow where:

1. Android receives an incoming call request.
2. The user accepts the call.
3. The backend creates an automatic conversation call session.
4. AI speaks first.
5. Android detects user speech automatically.
6. Android uploads utterance-level audio files.
7. The backend processes STT, AI response, and TTS.
8. Android plays AI audio and keeps listening for barge-in.
9. The call ends by user intent, timeout, or server command.
10. Android shows the conversation summary.

This is a documentation-level API spec. Backend Swagger and `contract/` files should be synchronized after implementation details are finalized.

## 2. Base Rules

### 2.1 Base URL

Local backend:

```http
http://localhost:3000
```

Android emulator:

```http
http://10.0.2.2:3000
```

### 2.2 Authentication

ver0.2 MVP does not require authentication.

Later versions should add authentication before exposing these APIs to production traffic.

### 2.3 Content Types

JSON APIs:

```http
Content-Type: application/json
```

Audio upload APIs:

```http
Content-Type: multipart/form-data
```

### 2.4 Time Format

All timestamps use ISO 8601 strings.

Example:

```json
"2026-05-30T06:00:00.000Z"
```

### 2.5 ID Rules

- Server IDs are generated by the backend.
- Android must generate `clientTurnId` for every automatic audio turn.
- `clientTurnId` must be unique within a call session.
- Reusing a completed `clientTurnId` should return the already-created result when possible.
- Reusing an in-progress `clientTurnId` should return `409 Conflict`.

  | "UPLOADED"
  | "TRANSCRIBING"
  | "TRANSCRIBED"
  | "RESPONDING"
  | "RESPONDED"
  | "SYNTHESIZING"
  | "COMPLETED"
  | "FAILED_STT"
  | "FAILED_LLM"
  | "FAILED_TTS"
  | "FAILED_UNKNOWN";

### 3.5 ConversationStep

```ts
type ConversationStep =
  | "greeting"
  | "wellbeing"
  | "meal"
  | "health"
  | "medication"
  | "sleep"
  | "schedule"
  | "mood"
  | "free_talk"
  | "closing";
```

Default target flow:

1. `wellbeing`: 오늘 하루 안부
2. `meal`: 식사 확인
3. `health`: 건강 확인
4. `medication`: 복약 확인
5. `sleep` or `schedule`: 수면/일정 확인
6. `mood`: 기분 확인
7. `free_talk`: 사용자가 더 대화하고 싶어 하는 경우

### 3.6 AutoTurnNextAction

```ts
type AutoTurnNextAction =
  | "play_audio"
  | "listen_again"
  | "end_call_after_audio"
  | "force_end";
```

Meaning:

| Value | Description |
| --- | --- |
| `play_audio` | Android should play `audioBase64`, then return to automatic listening. |
| `listen_again` | Android should play retry guidance if provided, then listen again. |
| `end_call_after_audio` | Android should play `audioBase64`, then end the call. |
| `force_end` | Android should end the call immediately. |

## 4. Common Models

### 4.1 AudioPolicy

Server may return this policy when a session is created. Android can start with hardcoded defaults, but server-provided values should take priority when present.

```json
{
  "sampleRateHz": 16000,
  "preferredFormat": "audio/mp4",
  "minSpeechMs": 500,
  "speechEndSilenceMs": 1300,
  "noResponsePromptMs": 3000,
  "maxUtteranceMs": 12000,
  "bargeInEnabled": true
}
```

Field rules:

| Field | Description |
| --- | --- |
| `sampleRateHz` | Preferred recording sample rate. |
| `preferredFormat` | Preferred upload audio format. Initial value can remain `audio/mp4` or `audio/m4a`. |
| `minSpeechMs` | Audio shorter than this can be ignored by Android. |
| `speechEndSilenceMs` | Silence duration used to close a user utterance. |
| `noResponsePromptMs` | Silence duration after AI question before Android asks again. |
| `maxUtteranceMs` | Maximum single user utterance length. |
| `bargeInEnabled` | If true, Android keeps listening during AI playback. |

### 4.2 ConversationPolicy

```json
{
  "maxDurationMs": 600000,
  "targetTurnCount": 5,
  "firstGreetingText": "안녕하세요 왕송길 어르신 AI통화 서비스 세요입니다!",
  "noResponsePromptText": "여보세요? 제 말 들리세요?",
  "maxDurationClosingText": "어르신 아쉽지만 오늘 통화는 여기까지에요."
}
```

### 4.3 ApiError

```json
{
  "statusCode": 400,
  "message": "Invalid request body",
  "error": "Bad Request",
  "code": "INVALID_REQUEST"
}
```

Recommended error codes:

| HTTP | Code | Description |
| --- | --- | --- |
| 400 | `INVALID_REQUEST` | Required body, field, or multipart data is invalid. |
| 404 | `NOT_FOUND` | Invitation, session, or turn does not exist. |
| 409 | `INVALID_STATE` | The resource exists but cannot accept the requested transition. |
| 409 | `TURN_ALREADY_PROCESSING` | Same `clientTurnId` is already being processed. |
| 413 | `AUDIO_TOO_LARGE` | Uploaded audio exceeds allowed size. |
| 415 | `UNSUPPORTED_AUDIO_TYPE` | Audio MIME type is unsupported. |
| 502 | `STT_FAILED` | Speech-to-text processing failed. |
| 502 | `LLM_FAILED` | AI response generation failed. |
| 502 | `TTS_FAILED` | Text-to-speech generation failed. |

Empty STT result is not treated as a hard failure in ver0.2. The backend should create a retry guidance response such as "다시 한 번 말씀해 주세요." and return `nextAction: "listen_again"`.

Creates a call session. ver0.2 automatic conversation must use `mode: "auto_conversation"`.

```http
POST /call-sessions
```

Request:

```json
{
  "mode": "auto_conversation",
  "source": "incoming_call",
  "callInvitationId": "cminv_001",
  "clientSessionId": "android-session-001"
}
```

Fields:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `mode` | string | no | Defaults to `manual_recording` for backward compatibility. Use `auto_conversation` for ver0.2. |
| `source` | string | no | Example: `incoming_call`, `android_test_button`, `manual_start`. |
| `callInvitationId` | string | no | Accepted invitation ID. |
| `clientSessionId` | string | no | Android-side debug/session correlation ID. |

  "audioPolicy": {
    "sampleRateHz": 16000,
    "preferredFormat": "audio/mp4",
    "minSpeechMs": 500,
    "speechEndSilenceMs": 1300,
    "noResponsePromptMs": 3000,
    "maxUtteranceMs": 12000,
    "bargeInEnabled": true
  },
  "conversationPolicy": {
    "maxDurationMs": 600000,
    "targetTurnCount": 5,
    "firstGreetingText": "안녕하세요 왕송길 어르신 AI통화 서비스 세요입니다!",
    "noResponsePromptText": "여보세요? 제 말 들리세요?",

Android behavior:

1. Move to call screen.
2. Play fixed AI greeting.
3. Start automatic listening.
4. Connect to `/call-sessions` Socket.IO namespace with this `sessionId`.

## 5.8 Upload Automatic Conversation Audio Turn

Core ver0.2 endpoint.

```http
POST /call-sessions/:id/auto-turns/audio
Content-Type: multipart/form-data
```

Multipart fields:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `audio` | file | yes | User utterance audio file. |
| `clientTurnId` | string | yes | Android-generated unique turn ID. |
| `startedAt` | ISO string | no | User speech start timestamp. |
| `endedAt` | ISO string | no | User speech end timestamp. |
| `durationMs` | number | no | Audio duration in milliseconds. |
| `mimeType` | string | no | Android audio MIME type. |
| `mode` | string | yes | Must be `auto_conversation`. |
| `bargeIn` | boolean | no | True if user interrupted AI playback. |
| `conversationStep` | string | no | Android's current expected step. Backend may override. |
| `clientSequence` | number | no | Monotonic Android turn sequence for debugging. |

Example request fields:

```text
audio=@utterance-001.m4a
clientTurnId=android-turn-001
startedAt=2026-05-30T06:00:22.000Z
endedAt=2026-05-30T06:00:25.000Z
durationMs=3000
mimeType=audio/mp4
mode=auto_conversation
bargeIn=false
conversationStep=wellbeing
clientSequence=1
```

Response `201 Created`:

```json
{
  "turnId": "cmturn_003",
  "clientTurnId": "android-turn-001",
  "sessionId": "cmsess_001",
  "status": "COMPLETED",
  "userText": "오늘은 괜찮았어요.",
  "assistantText": "괜찮으셨다니 다행이에요. 식사는 잘 챙겨 드셨어요?",
  "audioMimeType": "audio/mpeg",
  "audioBase64": "base64-encoded-audio",
  "conversationStep": "meal",
  "nextAction": "play_audio",
  "failed": false,
  "riskFlag": false,
  "riskType": null,
  "createdAt": "2026-05-30T06:00:26.000Z"
}
```

Empty STT response example:

```json
{
  "turnId": "cmturn_004",
  "clientTurnId": "android-turn-002",
  "sessionId": "cmsess_001",
  "status": "COMPLETED",
  "userText": "",
  "assistantText": "죄송해요, 잘 못 들었어요. 다시 한 번 말씀해 주세요.",
  "audioMimeType": "audio/mpeg",
  "audioBase64": "base64-encoded-audio",
  "conversationStep": "meal",
  "nextAction": "listen_again",
  "failed": false,
  "riskFlag": false,
  "riskType": null
}
```

User requested call end example:

```json
{
  "turnId": "cmturn_010",
  "clientTurnId": "android-turn-008",
  "sessionId": "cmsess_001",
  "status": "COMPLETED",
  "userText": "이제 끊어.",
  "assistantText": "네 어르신, 오늘 말씀 나눠주셔서 고마워요. 이제 통화 마칠게요.",
  "audioMimeType": "audio/mpeg",
  "audioBase64": "base64-encoded-audio",
  "conversationStep": "closing",
  "nextAction": "end_call_after_audio",
  "failed": false,
  "riskFlag": false,
  "riskType": null
}
```

Barge-in behavior:

- Android sets `bargeIn=true` when the user speaks while AI audio is playing.
- Android must stop AI playback immediately before uploading the user utterance.
- Backend may use `bargeIn=true` as prompt context, but should still process the turn normally.

Idempotency:

- If the same `clientTurnId` already completed, return the stored response.
- If the same `clientTurnId` is currently processing, return `409 TURN_ALREADY_PROCESSING`.
- If the session is `ENDED` or `EXPIRED`, return `409 INVALID_STATE`.

## 7. Backend State Transitions

### 7.1 Session Creation

```text
POST /call-sessions
  -> ACTIVE
  -> AI greeting turn stored
  -> WAITING_FOR_USER after Android starts listening
```

The first greeting may be stored as an assistant turn immediately when the session is created.

### 7.2 Automatic Turn Upload

```text
WAITING_FOR_USER or AI_SPEAKING
  -> POST /auto-turns/audio
  -> PROCESSING_TURN
  -> UPLOADED
  -> TRANSCRIBING
  -> TRANSCRIBED
  -> RESPONDING
  -> RESPONDED
  -> SYNTHESIZING
  -> COMPLETED
  -> AI_SPEAKING
  -> WAITING_FOR_USER
```

If `nextAction` is `end_call_after_audio`, backend should move session toward `ENDING` and then `ENDED` after Android acknowledges or after a timeout.

### 7.3 Failure Handling

STT empty:

```text
TRANSCRIBED with userText=""
  -> assistant retry text
  -> TTS
  -> COMPLETED
  -> nextAction=listen_again
```

STT/LLM/TTS hard failure:

```text
FAILED_STT | FAILED_LLM | FAILED_TTS
  -> turn_failed socket event
  -> HTTP 502 if response cannot be recovered
```

For demo stability, backend may return a fallback assistant response when LLM or TTS fails, but the failure type must still be logged.

## 8. Android API Responsibilities

Android must:

- Use `POST /call-invitations/test` only for test/demo call triggering.
- Accept or decline an invitation before creating a session.
- Create automatic call sessions with `mode: "auto_conversation"`.
- Play AI greeting after session creation.
- Keep microphone monitoring active during AI playback.
- Stop AI playback when user barge-in is detected.
- Upload one user utterance per `clientTurnId`.
- Treat `nextAction` as the source of truth for next UI/audio behavior.
- Call `POST /call-sessions/:id/end` when the user presses end call.
- Show conversation history during the call.
- Show summary screen after the call using `GET /call-sessions/:id/summary` or `GET /call-sessions/:id/turns`.

## 9. Backend API Responsibilities

Backend must:

- Persist session mode and session status.
- Persist turn status and `clientTurnId`.
- Deduplicate turns by `(sessionId, clientTurnId)`.
- Keep the default 5-step conversation policy.
- Detect user intent to end the call.
- Detect risk signals and log them first.
- Return retry guidance when transcription is empty.
- Distinguish STT, LLM, and TTS failures.
- Emit Socket.IO turn/session events.
- Enforce max session duration of 10 minutes.
- Provide enough data for Android post-call summary.

## 12. Acceptance Criteria

The ver0.2 API flow is acceptable when:

1. Android can trigger a test incoming call.
2. Android can accept the call and create an `auto_conversation` session.
3. AI greeting plays without the user pressing a recording button.
4. Android detects speech and uploads audio automatically.
5. Backend returns STT text, assistant text, and TTS audio.
6. Android plays the AI response automatically.
7. User can interrupt AI playback by speaking.
8. Five core conversation turns can complete without manual recording controls.
9. User can end the call by button or voice intent.
10. A call summary/conversation history screen is available after ending.
11. Server logs show session creation, turn processing, assistant answer, risk flag when present, and session end.
